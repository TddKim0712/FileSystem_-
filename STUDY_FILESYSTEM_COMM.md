## 1) 학습 방법론: 수동 독서에서 능동 설계로

### 1.1 이 문서를 읽는 태도

저수준 시스템 학습은 "개념 암기"보다 "제약 하에서의 선택"이 핵심입니다.
다음 질문을 반복하며 읽으세요.

- 내 시스템의 **최악 조건**은 무엇인가? (전원 차단, 온도, 진동, EMI, 방사선 등)
- 내 시스템의 **절대 제약**은 무엇인가? (RAM, ROM, 쓰기 수명, 지연 허용치)
- 내 시스템의 **성공 지표**는 무엇인가? (데이터 무결성, 부팅시간, 회복시간, 유지보수성)

### 1.2 능동 학습 루프

1. 읽기: 구조 이해
2. 도식화: 계층 다이어그램 손그림
3. 수치화: 블록 크기/페이지 크기/버퍼 개수 등
4. 실패 주입: "중간에 전원 꺼지면?" 질문
5. 재설계: 정책 수정

이 루프를 3회 반복하면 "아는 상태"에서 "설계 가능한 상태"로 넘어갑니다.

## 2) 저장장치와 데이터 표현의 물리적 현실

### 2.1 블록 디바이스 vs Raw Flash

#### 블록 디바이스

운영체제/파일시스템에게 `read(block)` / `write(block)` 추상화를 제공합니다.
내부 웨어레벨링/배드블록 관리는 컨트롤러가 숨기는 경우가 많습니다.

예:
- HDD
- SATA SSD
- eMMC/SD(내부 FTL 존재)

#### Raw Flash

파일시스템이 매체의 물리 제약을 직접 맞춰야 합니다.

주요 제약:
- erase-before-write
- 페이지 쓰기 정렬
- 블록 소거 단위
- 소거 횟수 제한(P/E cycle)

예:
- NOR/NAND 직결 MCU

### 2.2 저장 관련 4대 단위

- **섹터/블록**: 논리적 I/O 단위
- **페이지**: 플래시 쓰기 단위
- **소거 블록(Erase Block)**: 소거 단위
- **클러스터**: 파일시스템 할당 단위

설계 실수의 대표:
"파일시스템 클러스터 크기"를 "물리 페이지/소거블록"과 무관하게 선택해 쓰기 증폭을 유발.

### 2.3 무결성과 내구성의 구분

- **무결성(Integrity)**: 데이터가 올바른가?
- **내구성(Durability)**: 전원 차단 후에도 유지되는가?

CRC는 무결성 검사에 도움을 주지만,
내구성을 자동 보장하지는 않습니다.
내구성은 기록 순서, flush 정책, 원자적 갱신 설계가 좌우합니다.

## 3) FAT 계열의 역사적 진화

### 3.1 시대 배경

FAT는 "성능 최적"보다 "호환성과 구현 단순성"을 우선한 결과물입니다.
플로피 시대에는 단순 구조가 생존 전략이었습니다.

- FAT12: 플로피 중심의 소용량
- FAT16: HDD 확장 대응
- FAT32: 대용량 볼륨과 루트 디렉토리 한계 완화

### 3.2 왜 FAT가 오래 살아남았는가

1. 단순 구현: 부트로더/펌웨어에서도 구현 가능
2. 생태계 호환: 운영체제/복구툴/장치 제조사 지원
3. 이식성: 다양한 CPU/컴파일러/도구체인에서 동작

"기술적으로 최선"이 아니라 "시스템 전체 비용 최소"가 채택 이유인 경우가 많습니다.

## 4) FAT12/16/32 내부 구조 완전 해부

### 4.1 볼륨 레이아웃

일반적으로 다음 순서:

1. Reserved 영역 (Boot Sector + BPB)
2. FAT #1
3. FAT #2(복제본)
4. Root Directory(FAT12/16의 경우 고정)
5. Data 영역

### 4.2 BPB(BIOS Parameter Block)의 의미

BPB는 "이 볼륨을 어떻게 해석할지"를 규정하는 계약서입니다.

대표 필드 예:
- bytes per sector
- sectors per cluster
- reserved sectors
- number of FATs
- root entry count(FAT12/16)
- sectors per FAT

BPB 해석 오류는 전체 파일시스템 오인식으로 이어집니다.

### 4.3 FAT 엔트리 해석

#### FAT12
- 12비트 packed entry
- 엔트리 경계가 바이트 경계를 깔끔히 맞추지 않아 구현 난이도 증가

#### FAT16
- 16비트 entry
- FAT12보다 구현 단순

#### FAT32
- 32비트 슬롯 사용(실사용은 하위 유효비트 위주)
- 더 큰 클러스터 번호 공간

### 4.4 디렉토리 엔트리

전통 8.3 파일명, 속성 비트, 타임스탬프, 시작 클러스터, 파일 크기 정보를 갖습니다.
롱파일명(LFN)은 별도 특수 엔트리 연속으로 표현됩니다.

### 4.5 FAT 체인과 랜덤 접근

파일 데이터는 클러스터 체인으로 연결됩니다.

- 장점: 할당 구조 단순
- 단점: 긴 파일의 중간 접근 시 체인 탐색 비용

캐시 정책이 없으면 성능이 급락할 수 있습니다.

### 4.6 FAT 타입 판정의 실무 포인트

이름("FAT16")만 보고 고정하면 안 되고,
총 클러스터 수와 필드 조합으로 판정해야 합니다.
실장 시 스펙 임계값을 코드에 명시하고 단위테스트로 묶으세요.

## 5) FAT 엔지니어링: 성능, 단편화, 복구, 전원장애

### 5.1 내부 단편화

클러스터가 클수록 작은 파일이 낭비하는 공간이 커집니다.
특히 로그성 소파일 다량 생성 시스템에서 효율이 크게 떨어집니다.

### 5.2 외부 단편화(프래그먼트)

연속 공간이 부족하면 체인이 분절되어 탐색/읽기 비용이 상승합니다.
임베디드에서 회전디스크는 아니어도, 캐시 미스/메타데이터 접근 증가로 지연이 늘어납니다.

### 5.3 메타데이터 업데이트 순서 위험

파일 생성/확장 시 다음 업데이트가 연속 발생합니다.

1. 데이터 클러스터 기록
2. FAT 체인 갱신
3. 디렉토리 엔트리 갱신

중간 전원 차단 시 "고아 클러스터" 혹은 "크기 불일치" 가능.
FAT 자체는 저널링이 없으므로 fsck류 복구 도구 의존이 큽니다.

### 5.4 FAT 운용 팁

- 중요 데이터는 append-only 규칙 고려
- 주기적 동기화 지점(checkpoint) 설정
- 메타데이터 캐시 정책 문서화
- 테스트에서 강제 전원 차단 주입

## 6) littlefs: 플래시 시대의 파일시스템 설계 철학

### 6.1 littlefs의 문제 정의

"작은 MCU 환경에서 전원 차단에 강하고,
raw flash 제약(소거 단위/수명)을 고려하는 파일시스템이 필요하다."

### 6.2 핵심 설계 특성

- power-loss resilience 지향
- copy-on-write 성격의 갱신 패턴
- 메타데이터 pair 구조를 통한 원자성 보조
- 작은 메모리 풋프린트

### 6.3 장점

- 갑작스런 리셋/전원다운 상황에서 복구성 우수
- 임베디드 친화적 API/구성
- raw flash 제약과 궁합이 좋음

### 6.4 주의점

- PC 범용 호환성이 FAT보다 낮음
- 개발팀이 내부 동작(garbage collection/metadata layout)을 이해해야 운영 리스크가 줄어듦

### 6.5 littlefs 도입 체크리스트

- 전원 장애 빈도/리스크가 높은가?
- raw flash 직접 제어가 필요한가?
- 오프라인 PC 마운트 호환성이 필수인가?
- 파일 수/디렉토리 깊이/쓰기 패턴은 어떤가?

## 7) ext/ext2: Unix 계열 메타데이터 모델의 정석

### 7.1 ext 계열 맥락

- ext: 초기 세대
- ext2: 구조 완성도 높은 고전
- ext3/ext4: 저널링·확장성 개선

### 7.2 ext2의 핵심 구조

- superblock: 파일시스템 전체 메타
- block group: 지역성/관리 단위
- inode table: 파일 메타데이터
- directory entry: 이름→inode 매핑
- data block: 실제 데이터

### 7.3 FAT와의 철학 차이

- FAT: 전역 FAT 체인 중심
- ext2: inode 기반 객체 모델

ext2는 권한, 링크, 타임스탬프, 다양한 메타표현에 강합니다.
대신 단순 부트로더 수준 구현은 FAT보다 어렵습니다.

### 7.4 ext2의 약점과 운영 상쇄

ext2 자체는 저널링이 없어 비정상 종료 후 검사 비용이 증가합니다.
운영에서는 다음으로 상쇄합니다.

- 정전 대비 UPS/전원 설계
- fsck 정책
- 쓰기 경로 제어 및 동기화 정책

## 8) FAT vs littlefs vs ext2: 설계 의사결정 프레임워크

### 8.1 한 줄 요약

- 호환성이 1순위면 FAT
- raw flash + 전원안정성 중심이면 littlefs
- Unix 기능/대형 시스템이면 ext2 이상

### 8.2 비교표

| 항목 | FAT | littlefs | ext2 |
|---|---|---|---|
| 주요 장점 | 호환성/단순성 | 전원차단 내성/플래시 친화 | 메타데이터 모델/확장성 |
| 주요 약점 | 저널링 부재/단편화 | 범용 호환성 낮음 | 저널링 부재/복잡도 |
| 권장 매체 | SD/eMMC/호환매체 | NOR/NAND raw flash | Linux block device |
| 구현 난이도 | 낮음 | 중간 | 중~높음 |

### 8.3 결정 트리(간소)

1. PC에서 바로 읽어야 하는가? → 예: FAT 우선
2. raw flash 직결인가? → 예: littlefs 우선
3. Unix 권한/링크 모델 필요한가? → 예: ext 계열

## 9) OSI 1·2계층 핵심 개념을 저장시스템과 연결하기

### 9.1 L1: Physical Layer

핵심 요소:
- 전기적 레벨
- 타이밍/클럭
- 물리 매체 특성

저장 I/O 영향:
- BER 상승은 재전송/지연 증가로 이어짐
- 케이블/배선 품질이 안정성 지배

### 9.2 L2: Data Link Layer

핵심 요소:
- 프레임 경계
- 주소화
- CRC 오류검출
- 재전송/흐름제어

저장 I/O 영향:
- 프레임 손실률이 높으면 write latency tail이 길어짐
- ACK/재전송 정책이 실시간성에 직접 영향

### 9.3 대표 링크와 관찰 포인트

#### UART
- 단순함 장점
- 프레이밍/재전송/체크섬은 상위 프로토콜 책임

#### SPI
- 고속 근거리
- 버스 공유/CS 제어 정책 중요

#### I2C
- 주소 기반 멀티드롭
- 노이즈/속도/버스 길이 제약에 민감

#### CAN
- 강한 오류 제어, 중재 메커니즘
- 제어 시스템에서 예측가능성 확보에 유리

### 9.4 파일시스템과 링크 계층의 경계

FS는 블록 추상화만 보지만,
실제 시스템에서 지연/오류 패턴은 아래 계층이 결정합니다.

따라서 스펙 문서에 반드시 포함:
- 최대 재전송 횟수
- 타임아웃
- CRC 실패 시 정책
- 블록 크기와 프레임 크기 매핑

## 10) 주변기기와의 관계: 버스, 컨트롤러, 드라이버, DMA, 인터럽트

### 10.1 계층 그림

1. 애플리케이션
2. 파일시스템
3. 블록 계층/캐시
4. 디바이스 드라이버
5. 컨트롤러(DMA/IRQ)
6. 버스(SPI/I2C/CAN/PCIe 등)
7. 물리 매체

병목은 보통 "예상 밖의 경계"에서 생깁니다.
예: FS보다 DMA 버퍼 정렬, 혹은 IRQ 우선순위.

### 10.2 DMA의 의미

CPU가 바이트 복사에 소모되는 시간을 줄여주지만,
버퍼 정렬/캐시 일관성/동기화 문제가 생깁니다.

체크리스트:
- DMA 가능한 메모리 영역 구분
- 캐시 flush/invalidate 타이밍
- ISR에서 최소 작업 원칙

### 10.3 인터럽트와 저장 지연

잘못 설계된 ISR은 파일 I/O jitter를 키웁니다.
실시간성 요구가 있으면:
- ISR 최소화
- 하프/풀 버퍼 전략
- 작업 큐로 하위 처리 분리

### 10.4 FTL이 있는 장치의 함정

SD/eMMC는 블록 장치처럼 보이지만,
내부에서 GC/웨어레벨링/맵 갱신을 수행합니다.
그 결과 장시간 tail latency spike가 발생할 수 있습니다.

테스트는 평균값이 아니라 P99/P999를 봐야 합니다.

## 11) BIOS/부트/저수준 호출의 실제 의미

### 11.1 부트 체인

전원 인가 후 소프트웨어는 일반적으로 다음 흐름:

1. 초기 펌웨어(BIOS/ROM monitor)
2. 부트 매체 섹터 읽기
3. 1단계 로더 실행
4. 다음 단계 로더/커널 이미지 로드

### 11.2 BIOS INT 호출의 역할(고전 x86)

초기 로더는 드라이버가 없으므로 BIOS 서비스 의존도가 큽니다.
즉 "저수준 호출"은 단순 API가 아니라,
**OS가 올라오기 전 세계의 유일한 I/O 경로**라는 의미를 가집니다.

### 11.3 부트로더 설계 실무 포인트

- 코드 크기 제한 하에서 파서 단순화
- 예외 경로(읽기 실패/재시도) 최소한이라도 정의
- 포맷/버전 호환성 전략 수립

## 12) malloc과 메모리 할당 전략: 실시간·저신뢰·장시간 운영 관점

### 12.1 왜 malloc이 문제가 되는가

문제는 "속도"보다 "예측가능성"입니다.

- 단편화 누적
- worst-case 지연 불명확
- 장시간 운용 후 실패 시점 예측 어려움

### 12.2 대안 전략

- 정적 할당(static buffers)
- 고정 크기 메모리 풀
- slab 유사 할당기
- 링버퍼(스트리밍 I/O)

### 12.3 설계 원칙

- ISR 경로 동적할당 금지
- I/O 요청 큐 길이 상수화
- "최대 동시 요청" 수식 문서화
- OOM 동작 정의(드랍/재시도/리셋)

### 12.4 최소 계산 템플릿

총 RAM 사용량 =
- 코드/정적 데이터
- FS 캐시
- DMA 버퍼
- 프로토콜 버퍼
- 스택(태스크별)
- 안전 마진

이 값을 worst-case로 계산해야 합니다.

## 부록 C: FAT12/16/32 수학적/정량적 분석 노트

### C.1 주요 계산식

아래 식은 FAT 계열 분석 시 반복적으로 사용됩니다.

- `RootDirSectors = ((RootEntCnt * 32) + (BytsPerSec - 1)) // BytsPerSec`
- `FATSz = (FATSz16 != 0) ? FATSz16 : FATSz32`
- `DataSectors = TotSec - (RsvdSecCnt + (NumFATs * FATSz) + RootDirSectors)`
- `CountOfClusters = DataSectors // SecPerClus`

판정(개략):
- CountOfClusters가 작으면 FAT12
- 중간이면 FAT16
- 크면 FAT32

### C.2 내부 단편화 기대값 직관

파일 크기가 1~N 균등분포라고 가정하면,
클러스터 크기 C에서 평균 낭비는 대략 C/2에 가까워집니다.

실무적으로는 파일 크기 분포가 heavy-tail이므로,
"작은 파일 비중"이 크면 클러스터 증대의 손해가 급격히 커집니다.

### C.3 FAT 테이블 메모리 압박

FAT 캐시를 RAM에 얼마나 둘지 결정할 때,

- 엔트리 수 증가
- 랜덤 접근 빈도
- 캐시 miss 비용

을 함께 계산해야 합니다.

단순히 "볼륨이 크다"가 아니라,
"작업셋의 클러스터 지역성"이 성능을 지배합니다.

## 부록 E: ext2 심화 구조 메모

### E.1 inode 포인터 계층

전통 inode는
- direct block pointers
- single indirect
- double indirect
- triple indirect

구조를 통해 큰 파일을 주소화합니다.

이 구조는 단순 contiguous 할당보다 유연하지만,
메타데이터 접근 수가 증가할 수 있습니다.

### E.2 block group의 의미

block group은 공간 지역성을 높이고,
메타데이터/데이터를 분산 배치해 확장성과 복구 편의성을 제공합니다.

### E.3 fsck 비용 직관

저널링이 없는 환경에서 일관성 검사는
"볼륨 크기"와 "메타데이터 복잡도"의 영향을 강하게 받습니다.
운영 정책에서 부팅시간 SLA를 함께 고려해야 합니다.

## 부록 F: 통신(L1/L2) 정량 모델 초안

### F.1 유효 처리량 근사

`EffectiveThroughput ≈ RawRate * (1 - Overhead) * (1 - RetransmissionLoss)`

여기서 Overhead에는 프레임 헤더/CRC/ACK 대기,
RetransmissionLoss에는 에러율 기반 재전송 비용이 반영됩니다.

### F.2 블록 I/O 지연 근사

`T_block ≈ N_frames * (T_frame + T_ack + T_queue) + T_retry`

- `N_frames`: 블록 1개를 나누는 프레임 수
- `T_retry`: 에러 발생 시 추가 지연

프레임이 작을수록 `N_frames`가 커져 오버헤드가 증가합니다.

### F.3 tail latency 관리

평균보다 중요한 값:
- P95
- P99
- P999

실시간/고신뢰 시스템은 "가끔 매우 느린 요청"이 장애 원인이 됩니다.

## 26) 용어집 (Glossary)

- **BPB**: FAT 볼륨 파라미터를 담은 부트 섹터 내 구조
- **Cluster**: 파일시스템의 할당 단위
- **FTL**: Flash Translation Layer, flash를 블록 디바이스처럼 보이게 함
- **Wear Leveling**: 소거/쓰기 수명을 균등화하는 기법
- **Journaling**: 메타데이터/데이터 갱신 의도를 로그로 남겨 복구성 향상
- **ISR**: 인터럽트 서비스 루틴
- **DMA**: CPU 개입을 줄여 메모리↔장치 전송 수행
- **P99 latency**: 요청의 99%가 이 값 이하로 끝나는 지연 지표
- **Power-cut resilience**: 전원 차단 후 일관 상태 복구 능력
- **FMEA**: 실패모드 및 영향 분석
