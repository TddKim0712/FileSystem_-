## 2) 저장장치 물리와 논리 추상화

### 2.1 Block Device vs Raw Flash

#### Block Device

상위에는 `read(block)`/`write(block)`를 제공합니다.
내부의 배드블록 관리, 웨어레벨링, 매핑은 컨트롤러/FTL이 처리합니다.

예: SD/eMMC/SATA SSD

#### Raw Flash

파일시스템 또는 FTL 유사 계층이 직접 제약을 다뤄야 합니다.

- erase-before-write
- 페이지 단위 쓰기
- 블록 단위 소거
- 제한된 P/E cycle

예: MCU 직결 NOR/NAND

### 2.2 단위 정리

- 섹터/블록: 논리 I/O 단위
- 페이지: 플래시 쓰기 단위
- 소거블록: 플래시 소거 단위
- 클러스터: 파일시스템 할당 단위

**설계 핵심**: 클러스터를 페이지/소거블록과 무관하게 잡으면 쓰기 증폭이 커집니다.

### 2.3 신뢰성 용어 정리

- 무결성(Integrity): 데이터가 올바른가?
- 내구성(Durability): 전원 차단 후에도 남는가?
- 가용성(Availability): 언제든 읽고 쓸 수 있는가?

CRC는 무결성에 도움을 주지만 내구성을 자동 보장하지 않습니다.

## 3) FAT12/16/32 역사: 왜 그렇게 진화했는가

### 3.1 FAT12

플로피 중심 시대의 산물입니다.
작은 볼륨, 단순한 구현, 제한된 자원 환경에서 효율을 추구했습니다.

### 3.2 FAT16

HDD 용량 증가에 맞춰 클러스터 주소 공간을 확대했습니다.
FAT12 대비 구현 단순성과 확장성을 동시에 잡았습니다.

### 3.3 FAT32

대용량 볼륨 대응이 핵심입니다.
루트 디렉토리를 고정영역에서 체인 기반으로 전환해 유연성을 높였습니다.

### 3.4 진화의 본질

FAT는 항상 다음 3가지 균형을 맞췄습니다.

1. 호환성
2. 구현 난이도
3. 저장장치 용량 확장

## 4) FAT 내부구조 정밀 해부

### 4.1 볼륨 레이아웃

전형적 구성:

1. Reserved Area (Boot Sector, BPB)
2. FAT #1
3. FAT #2 (미러)
4. Root Directory(FAT12/16)
5. Data Area

### 4.2 BPB 핵심 필드

- BytsPerSec
- SecPerClus
- RsvdSecCnt
- NumFATs
- RootEntCnt(FAT12/16)
- TotSec16/TotSec32
- FATSz16/FATSz32

BPB를 잘못 읽으면 전체 주소계산이 틀어집니다.

### 4.3 계산식 (실무용)

- `RootDirSectors = ((RootEntCnt * 32) + (BytsPerSec - 1)) // BytsPerSec`
- `FATSz = (FATSz16 != 0) ? FATSz16 : FATSz32`
- `DataSectors = TotSec - (RsvdSecCnt + NumFATs*FATSz + RootDirSectors)`
- `CountOfClusters = DataSectors // SecPerClus`

### 4.4 엔트리 비트폭 차이

- FAT12: 12비트 packed entry (파싱 난이도 높음)
- FAT16: 16비트 entry
- FAT32: 32비트 슬롯(실사용 유효비트 제한)

### 4.5 디렉토리 엔트리

- 파일명(8.3 + LFN 보조 엔트리)
- 속성
- 시작 클러스터
- 파일 크기
- 타임스탬프

### 4.6 체인 모델의 본질

FAT는 전역 체인 인덱스입니다.

장점:
- 구조 단순
- 디버깅 용이

약점:
- 랜덤 접근 시 체인 추적 비용
- 메타데이터 업데이트 취약성

## 5) FAT의 공학적 장단점과 실패모드

### 5.1 장점

- 구현 단순성 (부트로더/펌웨어에 적합)
- 호환성 압도적
- 도구 생태계 풍부

### 5.2 약점

- 저널링 부재
- 단편화
- 전원차단 시 메타데이터 불일치
- raw flash 친화성 부족

### 5.3 대표 실패모드

1. Lost cluster
2. Cross-linked file
3. 파일 크기-체인 길이 불일치
4. FAT 미러 불일치

### 5.4 완화 전략

- append-only 패턴
- 체크포인트 파일
- 주기적 무결성 검사
- 복구도구(fsck류) 정책화

---

## 6) littlefs 설계 철학과 운용 포인트

### 6.1 설계 목적

- power-loss resilience
- raw flash 제약 대응
- 작은 메모리 환경 지원

### 6.2 핵심 특징

- copy-on-write 성격
- 메타데이터 pair 구조
- wear leveling 고려

### 6.3 적합한 환경

- MCU + NOR/NAND raw flash
- 전원차단 리스크가 높은 장비
- 소형 메모리 예산

### 6.4 운영시 주의

- 파라미터(block/cache/lookahead) 튜닝 필요
- 성능은 평균이 아니라 tail latency로 판단
- power-cut 테스트를 설계 단계부터 포함

---

## 7) ext/ext2의 구조와 의미

### 7.1 ext 계열 맥락

- ext: 초기 세대
- ext2: 안정된 기본 구조
- ext3/ext4: 저널링/확장성 개선

### 7.2 ext2 핵심

- superblock
- block group
- inode table
- directory entry
- data block

### 7.3 inode 관점

FAT의 “클러스터 체인” 사고가 아니라,
ext2는 “객체(inode) + 포인터 계층” 사고입니다.

### 7.4 장단점

장점:
- 메타데이터 표현력 우수
- Unix semantics 강력

약점:
- 임베디드 초소형 환경에는 부담
- ext2 자체는 저널링 부재

## 8) FAT vs littlefs vs ext2 의사결정 프레임워크

| 항목 | FAT | littlefs | ext2 |
|---|---|---|---|
| 최우선 가치 | 호환성 | 전원내성+플래시 친화 | Unix 구조/확장성 |
| 권장 매체 | SD/eMMC/교환매체 | raw NOR/NAND | Linux block device |
| 구현 난이도 | 낮음 | 중간 | 중~높음 |
| 전원차단 내성 | 낮음(기본) | 높게 설계 | 낮음(ext2 기준) |

2. raw flash를 직접 다루는가?
3. Unix 권한/링크 모델이 필요한가?
4. 전원차단 위험이 높은가?

## 9) OSI 1·2계층 핵심과 저장 I/O 연결

### 9.1 L1 Physical

- 전압/클럭/배선/EMI
- BER(비트 오류율) 관리

### 9.2 L2 Data Link

- 프레임
- 주소
- CRC
- 재전송/흐름제어

### 9.3 저장경로와의 결합

블록 I/O는 상위에서 단순해 보이지만,
하위 L1/L2 오류와 재전송 정책이 지연/실패율을 결정합니다.

### 9.4 간단 모델

- `EffectiveThroughput ≈ RawRate * (1 - Overhead) * (1 - RetransmissionLoss)`
- `T_block ≈ N_frames*(T_frame + T_ack + T_queue) + T_retry`

## 10) 주변기기·드라이버·DMA·IRQ와 저장 경로

### 10.1 계층 경로

App → FS → Block Layer → Driver → Controller(DMA/IRQ) → Bus → Media

### 10.2 DMA

장점:
- CPU 오프로딩

주의:
- 버퍼 정렬
- 캐시 일관성
- 동기화

### 10.3 IRQ

ISR에서 무거운 작업을 하면 jitter가 커집니다.
원칙:
- ISR 최소화
- 하위 처리 분리

### 10.4 FTL 함정

SD/eMMC는 블록 장치처럼 보이지만 내부 GC로 tail latency spike가 생길 수 있습니다.
평균보다 P99/P999를 봐야 합니다.

## 11) BIOS/부트/저수준 호출의 실제 역할

### 11.1 고전 부트 체인

POST → 부트섹터 로드 → 1단계 로더 → 추가 로더/커널

### 11.2 BIOS 호출의 의미

초기 환경에서는 드라이버가 없기 때문에,
BIOS 호출이 사실상 유일한 I/O 경로입니다.

### 11.3 설계 포인트

- 코드 크기 제한
- 읽기 실패 재시도
- 헤더/BPB 검증

## 12) malloc/메모리 전략: 예측가능성 중심 설계

### 12.1 왜 malloc이 위험해지는가

- 단편화
- worst-case 지연 불확실
- 장시간 운용에서 실패시점 불명확

### 12.2 대안

- 정적 할당
- 고정 크기 풀
- slab 유사 구조
- 링버퍼

### 12.3 규칙

- ISR 경로 동적할당 금지
- 버퍼 수 상수화
- OOM 정책 사전 정의
